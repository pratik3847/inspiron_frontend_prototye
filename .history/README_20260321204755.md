# US Healthcare EDI Parser & X12 File Validator (837/835/834)

Domain: Healthcare IT / FinTech

## Overview
The US healthcare system processes billions of EDI transactions annually. The core X12 transaction sets are:
- 837P/837I: professional and institutional medical claims submitted by providers to payers
- 835: Electronic Remittance Advice sent back after claim adjudication
- 834: Benefit Enrollment and Maintenance for member add/change/terminate events

These ASCII-delimited files are strict, brittle, and costly to get wrong. Missing ISA/GS/ST segments, invalid qualifiers, or mismatched subscriber data can trigger claim rejections or enrollment failures that delay payments or coverage.

This project builds a full-featured, open-source X12 parser and validator for 837, 835, and 834 files with a clean web UI, AI-assisted error explanations, and actionable fix suggestions.

## Problem Statement
Build a web-based application that allows a user to upload any X12 837, 835, or 834 file, automatically detect the transaction type, parse the full loop/segment structure, validate it against the relevant X12 implementation guide rules, surface all errors with plain-language explanations, and suggest corrections. Provide an AI assistant for follow-up questions.

## Core Features
### Ingestion: File Upload and Transaction Detection
- Accept .edi, .txt, .dat, or .x12 uploads (drag-and-drop or file picker)
- Auto-detect ISA/GS/ST segments to identify 837P, 837I, 835, or 834
- Display envelope metadata: sender/receiver IDs, interchange date, GS functional group, transaction set count

### Parser Engine: Full Structural Parser
- Parse complete loop hierarchy per HIPAA implementation guides for 837, 835, and 834
- Render the parsed tree in an interactive, collapsible UI with element-level values and human-readable element names

### Validator: Validation Engine
- Required segment/element checks
- Element format validation: NPI (Luhn 10-digit), ZIP, dates (CCYYMMDD), monetary amounts
- Qualifier validation: NM108 entity codes, CLM05 facility type and frequency, SVC01 CPT/HCPCS format
- Cross-segment consistency: patient DOB vs claim date, total claim charge vs sum of service line charges
- 835-specific rules: CAS adjustment reasons, PR/CO/OA/PI group codes, CLP reconciliation
- 834-specific rules: INS maintenance type codes, member relationship codes, subscriber group/policy numbers, date consistency, duplicate member detection

### UX and AI: Error Report and Explanations
- List errors and warnings with loop location, segment ID, element position, error code, and plain-English description
- AI chat panel for contextual, EDI-specific questions

### Fix Assistant: Correction Suggestions
- Auto-suggest corrections for deterministic errors
- Allow applying fixes to the parsed structure with immediate re-validation

### 835 Feature: Remittance Summary
- Parse CLP loops and render claim ID, billed amount, paid amount, patient responsibility, adjustment reasons, and check/EFT reference

### 834 Feature: Member Enrollment Summary
- Parse Loop 2000 member records and display a summary table with color-coded maintenance types
- COB view, dependent rollup, and family enrollment grouping

## Key X12 Concepts (Quick Reference)
| Term | Meaning |
| --- | --- |
| ISA/IEA | Interchange Control Header/Trailer (outer envelope) |
| GS/GE | Functional Group Header/Trailer |
| ST/SE | Transaction Set Header/Trailer |
| Loop | Hierarchical grouping of segments (example: 2300 is one claim) |
| Segment | A line in the file, e.g., `CLM*1234*500***11:B:1*Y*A*Y*I~` |
| Element | Individual field within a segment, separated by `*` |
| NPI | National Provider Identifier (10-digit, Luhn check) |
| CARC/RARC | Claim/Remark Adjustment Reason Codes |
| 837P / 837I | Professional / Institutional claims |
| 835 | Electronic Remittance Advice |
| 834 | Benefit Enrollment and Maintenance |
| INS | 834 subscriber detail segment |
| HD | 834 coverage detail segment |
| DTP | Date/Time Period segment |
| REF | Reference ID segment (group/policy/SSN in 834) |

- Custom rule builder UI
- Export parsed JSON, error report PDF, 834 roster CSV, or corrected EDI

## Sample User Stories
- As a medical billing specialist, I want to upload an 837P and see validation errors immediately so I can fix them before submission.
- As a healthcare developer, I want to browse the parsed segment tree to understand structure without reading raw text.
- As a billing manager, I want an 835 summary showing paid, adjusted, or denied claims with reason codes in plain English.
- As a benefits administrator, I want to see adds/changes/terminations in an 834 file and any enrollment errors.
- As a new billing coder, I want to ask the AI assistant questions and get accurate explanations.
